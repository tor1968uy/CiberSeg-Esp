# 🛠️ Node.js en Windows: El peligro de la resolución de módulos "heredada"

Un reciente informe de la **Zero Day Initiative (ZDI)** ha puesto al descubierto una vulnerabilidad en el motor de **Node.js para Windows**. El problema radica en cómo Node.js busca y carga módulos dependientes, una debilidad que los atacantes pueden explotar para ejecutar código malicioso con privilegios elevados.

---

## 🔍 La Vulnerabilidad: "Trust Falls" en el Path

En entornos Windows, Node.js utiliza una lógica de búsqueda de rutas que, en ciertas configuraciones, puede ser manipulada. Si un atacante logra colocar un archivo malicioso en un directorio con permisos débiles que esté "antes" en el orden de búsqueda que el módulo legítimo, Node.js lo ejecutará sin cuestionarlo.

### El flujo del ataque:
1. **Búsqueda Prioritaria**: Node.js busca archivos necesarios en el directorio de trabajo actual (`CWD`) antes de ir a los directorios del sistema o de la aplicación.
2. **Inyección de binarios**: Un atacante con acceso limitado al sistema coloca una versión maliciosa de una dependencia (como un archivo `.exe` o una librería específica) en un directorio compartido.
3. **Ejecución**: Cuando un usuario con privilegios de administrador ejecuta un comando de Node.js desde ese directorio, el motor carga el binario del atacante, otorgándole control total sobre el sistema.



---

## 💻 ¿Por qué es un riesgo crítico?

* **Entornos de Desarrollo Compartidos**: Muy común en servidores de compilación (Build Servers) o máquinas donde varios desarrolladores comparten carpetas.
* **Scripts de Automatización**: Muchos administradores usan Node.js para automatizar tareas en Windows. Si estos scripts se ejecutan en carpetas "contaminadas", el compromiso es inmediato.
* **Silencioso**: No requiere que el usuario instale nada nuevo; simplemente aprovecha la confianza implícita de Node.js en su entorno local.

---

## 🛡️ Mitigación y Buenas Prácticas

La ZDI y la comunidad de Node.js recomiendan las siguientes acciones inmediatas:

1. **Actualización Urgente**: Instalar las últimas versiones de Node.js que corrigen el comportamiento de búsqueda en Windows.
2. **Evitar Ejecución en Directorios no Confiables**: Nunca ejecutes `npm install` o comandos de `node` en carpetas donde otros usuarios tengan permisos de escritura.
3. **Hardening del PATH**: Asegúrate de que las rutas del sistema en las variables de entorno estén protegidas y que el directorio actual (`.`) no tenga prioridad innecesaria.

---

## 🛠️ Tip de Auditoría: Verificando permisos en carpetas de Node

Puedes usar este comando de PowerShell para identificar carpetas de tus proyectos Node que tienen permisos de escritura para usuarios no administradores (un riesgo alto):

```powershell
# Listar permisos de la carpeta actual para detectar usuarios con permisos de escritura (W)
Get-Acl . | Select-Object -ExpandProperty Access | Where-Object { $_.FileSystemRights -match "Write" -and $_.IdentityReference -notmatch "Administrators" }
